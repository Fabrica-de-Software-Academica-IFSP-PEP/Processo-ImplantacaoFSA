Processo de Implantação de Software da FSA - IFSP/PEP

A1. Preparar Ambiente de Implantação

Descrição

Instalar as bibliotecas necessárias para funcionar a aplicação e/ou assegurar que o ambiente de disponibilização da aplicação esteja conforme o esperado de modo que seja um servidor apto à disponibilizar a aplicação ou um repositório para disponibilizar a aplicação via download.

Responsáveis e responsabilidades

Scrum Team

Entrada e Saída

Entrada:

  • Acesso ao servidor e/ou repositório.

Saída:

  • Ambiente configurado e pronto para uso.
  • Bibliotecas globais instaladas e configuradas.

Tarefas

  • A1.1 Instalar bibliotecas e pacotes necessários
  • A1.2 Instalar e configurar servidor
  • A1.3 Testar a disponibilidade do ambiente

Recursos

  • Servidor físico ou virtual (nuvem/local)
    • Servidor do IFSP - PEP
    • Digital Ocean
    • AWS

A2. Disponibilizar a Aplicação para Validação

Descrição

Instalar as dependências necessárias para garantir a aplicação conforme a última versão esperada, mantendo frontend, backend e banco de dados atualizados, realizar testes de verificação para assegurar o correto funcionamento do sistema com o mínimo de erros, e por fim construir e disponibilizar a solução para acesso para homologação.

Responsáveis e responsabilidades

Scrum Team / Pipeline

Entrada e Saída

Entrada:

  • Aplicação em um estado funcional.
  • Scripts de configuração e automação.
  • Banco de dados atualizado.
  • Ambiente de homologação disponível e configurado.

Saída:

  • Sistema implantado para homologação.
  • Aplicação acessível aos usuários.

Tarefas

  • A2.1 Instalar ou atualizar aplicação
  • A2.2 Instalar e atualizar dependências
  • A2.3 Instalar ou atualizar banco de dados se necessário
  • A2.4 Executar construção (build) do sistema
  • A2.5 Configurar ambiente de homologação
  • A2.6 Realizar testes de fumaça
  • A2.7 Executar o deploy para o ambiente de homologação

Recursos

  • Ferramentas de CI/CD, Pipelines
    • GitHub Actions
    • Jenkins
  • Imagens e Contêineres para Servidores
    • Docker
  • Sistema Gerenciador de Banco de Dados
    • MySQL
    • PostgreSQL

Materiais de apoio/referências

A4. Aplicar Testes de Aceitação

Descrição

O Scrum Team aplica os testes de aceitação durante a capacitação dos usuários, utilizando os cenários reais de uso do sistema. O objetivo é verificar se o software atende aos requisitos de negócio, identificar possíveis problemas e coletar feedbacks dos usuários enquanto eles aprendem a operar o sistema.

Responsáveis e responsabilidades

Scrum Team

Entradas e saídas

Entradas:

  • Plano de Verificação e Validação contendo critérios de aceite;
  • Casos de uso ou histórias de usuário aprovadas;
  • Scripts ou checklists de teste;
  • Ambiente de homologação configurado e baseline instalada;
  • Dados de teste;
  • Materiais de treinamento preparados.

Saídas:

  • Relatórios de testes com evidências de validação do sistema;
  • Feedback dos usuários sobre usabilidade e funcionalidades;
  • Lista de não conformidades registradas;
  • Documento formal de aceite.

Tarefas

  • 4.1 Preparar roteiro de testes integrando cenários de treinamento;
  • 4.2 Conduzir sessão de treinamento prática com usuários;
  • 4.3 Executar testes de aceitação simultaneamente ao treinamento;
  • 4.4 Registrar feedbacks, resultados e evidências;
  • 4.5 Consolidar relatório de validação e feedback;
  • 4.6 Encaminhar relatório e evidências para aprovação.

Recursos

  • Ferramentas de gestão de testes
    • TestRail
    • Zephyr for Jira
  • Ferramentas de registro de defeitos
    • Jira Software
    • Trello
    • Redmine
  • Software para captura de evidências
    • Lightshot
    • OBS Studio
  • Materiais de treinamento
  • Ambiente de homologação estável

A5. Entrega do Sistema

Descrição

A etapa de entrega do sistema marca a transição da solução validada em homologação para o ambiente de produção. O foco é garantir que a versão final esteja disponível para os usuários finais de forma segura, estável e com rastreabilidade. Essa fase não repete atividades já realizadas nos tópicos anteriores, mas concentra-se em coordenação, mitigação de riscos, execução do deploy definitivo, validação mínima e monitoramento pós-implantação.

Responsável e responsabilidades

Scrum Team

Entradas e saídas

Entradas:

  • Versão final validada em homologação e aprovada em testes de aceitação;
  • Plano de rollback aprovado;
  • Plano de comunicação aprovado;
  • Acesso ao ambiente de produção.

Saídas:

  • Sistema implantado, configurado, operacional e estável em produção;
  • Relatórios de implantação e evidências registradas;
  • Monitoramento inicial documentado.

Tarefas

  • 5.1 Planejar e comunicar a implantação junto às partes interessadas;
  • 5.2 Garantir mecanismos de contingência e backup do ambiente;
  • 5.3 Executar a implantação da versão final em produção;
  • 5.4 Validar a disponibilidade e funcionamento essencial do sistema;
  • 5.5 Monitorar o sistema após a implantação e registrar evidências.

Recursos

  • Ambiente de produção disponível;
  • Documentação de plano de comunicação e rollback;
  • Infraestrutura para backup e restauração;
  • Ferramentas de automação e deploy;
  • Checklists e materiais de validação;
  • Sistemas de monitoramento e registro de logs;
  • Documentação de implantação.
