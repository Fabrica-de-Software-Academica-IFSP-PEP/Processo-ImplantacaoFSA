Processo de Implantação de Software da FSA - IFSP/PEP

A1. Preparar Ambiente

Voltar para o diagrama

Descrição

Instalar as bibliotecas necessárias para funcionar a aplicação e/ou assegurar que o ambiente de disponibilização da aplicação esteja conforme o esperado de modo que seja um servidor apto à disponibilizar a aplicação ou um repositório para disponibilizar a aplicação via download.

Responsáveis e responsabilidades

Scrum Team

Entradas e Saídas

Entradas:

  • Acesso ao servidor ou repositório

Saídas:

  • Ambiente configurado
  • Bibliotecas globais instaladas

Tarefas

  • A1.1 Instalar bibliotecas e pacotes necessários
  • A1.2 Instalar e configurar servidor
  • A1.3 Testar a disponibilidade do ambiente

Recursos

  • Servidor físico ou virtual
    • Servidor Local do IFSP
    • AWS
    • DigitalOcean
  • Repositório para publicação
    • GitHub Releases
    • Drive

A2. Disponibilizar a aplicação para validação

Voltar para o diagrama

Descrição

Instalar as dependências necessárias para garantir a aplicação conforme a última versão esperada, mantendo frontend, backend e banco de dados atualizados, realizar testes de verificação para assegurar o correto funcionamento do sistema com o mínimo de erros, e por fim construir e disponibilizar a solução para acesso para homologação.

Responsáveis e responsabilidades

Scrum Team ou Pipeline

Entradas e Saídas

Entradas:

  • Aplicação em um estado funcional.
  • Scripts de configuração e automação.
  • Banco de dados atualizado.
  • Ambiente de homologação disponível e configurado.
  • Plano de rollback aprovado.

Saídas:

  • Sistema implantado para homologação.
  • Aplicação acessível aos usuários.

Tarefas

  • A2.1 Instalar ou atualizar aplicação
  • A2.2 Instalar e atualizar dependências
  • A2.3 Instalar ou atualizar banco de dados se necessário
  • A2.4 Executar construção (build) do sistema
  • A2.5 Garantir mecanismos de contingência e backup do ambiente
  • A2.6 Configurar ambiente de homologação
  • A2.7 Realizar testes de fumaça
  • A2.8 Executar o deploy para o ambiente de homologação

Recursos

  • Documentação de plano de comunicação e rollback.
  • Infraestrutura para backup e restauração.
  • Ferramentas de CI/CD, Pipelines
    • GitHub Actions
    • Jenkins
  • Imagens e Contêineres para Servidores
    • Docker
  • Sistema Gerenciador de Banco de Dados
    • MySQL
    • PostgreSQL

Materiais de apoio/referências

GITHUB DOCS. Guia de implantação contínua com GitHub Actions. GitHub, 2025. Disponível em: https://docs.github.com/pt/actions/deployment . Acesso em: 16 ago. 2025.

JENKINS. Pipeline de CI/CD: Guia de automação. Jenkins Documentation, 2025. Disponível em: https://www.jenkins.io/doc/book/pipeline/ . Acesso em: 16 ago. 2025.

DOCKER. Como criar ambientes de homologação com Docker. Docker Docs, 2025. Disponível em: https://docs.docker.com/get-started/overview/ . Acesso em: 16 ago. 2025.

A3. Preparar material de treinamento para os usuários, roteiro e cenários de testes

Voltar para o diagrama

Descrição

Organizar e disponibilizar conteúdos de apoio para capacitar os usuários no uso do sistema, bem como estruturar um roteiro de testes com cenários que integrem o aprendizado obtido no treinamento.

Responsáveis e responsabilidades

Scrum Team

Entradas e Saídas

Entradas:

  • Ambiente de homologação configurado e estável.

Saídas:

  • Cronograma de treinamento definido;
  • Materiais de apoio preparados;
  • Roteiro de testes pronto.

Tarefas

  • A3.1 Planejar cronograma de treinamento com base nas áreas envolvidas;
  • A3.2 Preparar e revisar os materiais de apoio(documentos, slides, guias, vídeos);
  • A3.3 Preparar roteiro de testes integrando cenários de treinamento.

Recursos

  • Materiais didáticos (documentação, manuais, vídeos tutoriais, FAQs);
  • Software de apoio para apresentações (PowerPoint, Google Slides);
  • Ambiente de teste ou homologação para execução dos cenários.

A4. Capacitar os usuários

Voltar para o diagrama

Descrição

Treinar os usuários do sistema a fim de que estejam aptos a realizarem as atividades funcionais e/ou não-funcionais da aplicação.

Responsável e responsabilidades

Scrum Team

Entradas e Saídas

Entradas:

  • Ambiente de homologação configurado e estável;
  • Agenda e cronograma de treinamento definidos;
  • Materiais de apoio preparados.

Saídas:

  • Usuários treinados e aptos a operar o sistema;
  • Materiais de apoio (manuais, tutoriais, FAQs, vídeos);
  • Materiais de treinamento distribuídos;
  • Registros de participação e presença;
  • Feedback dos usuários sobre clareza e efetividade do treinamento.

Tarefas

  • A4.1 Conduzir sessões de treinamento teóricas e práticas;
  • A4.2 Aplicar simulações de uso do sistema;
  • A4.3 Coletar dúvidas e feedbacks dos participantes;
  • A4.4 Elaborar relatório de capacitação contendo lista de participantes, avaliação de aprendizado e pontos de melhoria.

Recursos

  • Ambiente de homologação dedicado;
  • Materiais didáticos (documentação, manuais, vídeos tutoriais, FAQs);
  • Ferramentas de conferência online (Zoom, Teams, Google Meet) para treinamentos remotos se necessário;
  • Software de apoio para apresentações (PowerPoint, Google Slides);
  • Ferramentas de registro de presença e avaliação (Google Forms, Microsoft Forms, etc.).

A5. Aplicar testes de aceitação

Voltar para o diagrama

Descrição

O Scrum Team aplica os testes de aceitação durante a capacitação dos usuários, utilizando os cenários reais de uso do sistema. O objetivo é verificar se o software atende aos requisitos de negócio, identificar possíveis problemas e coletar feedbacks dos usuários enquanto eles aprendem a operar o sistema.

Responsável e responsabilidades

Scrum Team

Entradas e Saídas

Entradas:

  • Plano de Verificação e Validação contendo critérios de aceite;
  • Casos de uso ou histórias de usuário aprovadas;
  • Scripts ou checklists de teste;
  • Ambiente de homologação configurado e baseline instalada;
  • Dados de teste;
  • Materiais de treinamento preparados.

Saídas:

  • Relatórios de testes com evidências de validação do sistema;
  • Feedback dos usuários sobre usabilidade e funcionalidades;
  • Lista de não conformidades registradas;
  • Documento formal de aceite.

Tarefas

  • A5.1 Conduzir sessão de treinamento prática com usuários;
  • A5.2 Executar testes de aceitação simultaneamente ao treinamento;
  • A5.3 Registrar feedbacks, resultados e evidências;
  • A5.4 Consolidar relatório de validação e feedback;
  • A5.5 Encaminhar relatório e evidências para aprovação.

Recursos

  • Ferramentas de gestão de testes:
    • TestRail
    • Zephyr
  • Ferramentas de registro de defeitos:
    • Jira
    • Trello
    • Redmine
  • Software para captura de evidências:
    • Lihtshot
    • OBS Studio
  • Materiais de treinamento;
  • Ambiente de homologação estável.

6. Entregar Sistema em Produção

Voltar para o diagrama

Descrição

A etapa de entrega do sistema marca a transição da solução validada em homologação para o ambiente de produção. O foco é garantir que a versão final esteja disponível para os usuários finais de forma segura, estável e com rastreabilidade. Essa fase não repete atividades já realizadas nos tópicos anteriores, mas concentra-se em coordenação, mitigação de riscos, execução do deploy definitivo, validação mínima e monitoramento pós-implantação.

Responsável e responsabilidades

Scrum Team

Entradas e Saídas

Entradas:

  • Versão final validada em homologação e aprovada em testes de aceitação;
  • Plano de comunicação aprovado;
  • Acesso ao ambiente de produção.

Saídas:

  • Sistema implantado, configurado, operacional e estável em produção;
  • Relatórios de implantação e evidências registradas;
  • Monitoramento inicial documentado.

Tarefas

  • A6.1 Planejar e comunicar a implantação junto às partes interessadas;
  • A6.2 Executar a implantação da versão final em produção;
  • A6.3 Validar a disponibilidade e funcionamento essencial do sistema;
  • A6.4 Monitorar o sistema após a implantação e registrar evidências.

Recursos

  • Ambiente de produção disponível;
  • Ferramentas de automação e deploy;
  • Checklists e materiais de validação;
  • Sistemas de monitoramento e registro de logs;
  • Documentação de implantação.
